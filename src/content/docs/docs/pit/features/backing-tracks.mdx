> Half of modern musical-theatre productions run to a click and tracks. Backing tracks are how Pit becomes "yes, you can run that show on this."

## What ships in v0.13.0

Backing track support that covers the working case: per-song audio files, song transport, cue points, vamp interaction, and a transport-state machine the engine and UI both understand.

The full UX, transport-state-machine semantics, and vamp-transition rules get **refined during the v0.13.0 spec session** — the list below is the committed-to scope at the version-planning level.

## Audio decoding

[Symphonia](https://github.com/pdeljanov/Symphonia) (pure-Rust audio decoder) for:

- **mp3**
- **wav** (PCM, 16/24/32-bit)
- **flac**
- **ogg vorbis**

Sample rates and channel counts mismatched against the engine are converted on load (cubic resample) — not on the realtime path.

## Per-song tracks tab

Each song's settings get a new "Backing" sub-tab with a per-track row editor:

| Column | What |
|---|---|
| File | Path to audio file, drag-drop supported |
| Label | Display name in transport widget |
| Stem type | Drums / bass / guitars / keys / orch / vox / click / custom |
| Bus output | Which bus this track routes to (defaults vary by stem type) |
| Gain | Per-track gain trim (dB) |
| Pan | Per-track pan |
| Mute | Boolean |
| Start offset | Sample-accurate offset from song start |

A song can have any number of tracks. Common shape: 4–8 stems + a click.

## Song transport

A first-class engine concept (introduced in v0.8.0 for click, extended in v0.13.0 for tracks):

- **Play / Pause / Stop / Seek**
- **Position** (bar:beat or seconds)
- **Sample-accurate cue jump** — jump to a cue point without losing sync
- **MIDI clock sync** — clock send tracks the transport BPM

The transport panel in Program mode lets you scrub during authoring; in Perform mode, the transport widget exposes play/pause/jump plus the position display.

## Cue points within tracks

Cue points are named timeline anchors that the transport can jump to:

- Editable in the click track editor (v0.12.0) and the new backing-track editor
- Jump-to-cue via button/switch rig component (action: "jump to cue")
- Visible in the transport widget

## Patch transport triggers

Optional per-patch hooks:

- **onEnter** — when this patch becomes active, fire a transport command (play / jump-to-cue / etc.)
- **onExit** — when this patch becomes inactive, same

**Off by default.** When on, they make patch advances ↔ track positions a coupled system; this is powerful but error-prone in performance, so it's opt-in per patch.

## Vamp interaction

Vamps in the click track (authored in v0.12.0) are regions of time that loop. With backing tracks:

- A vamp region in the click track also loops the transport over the same region
- A **footswitch action "end vamp at next loop point"** marks the vamp as "exit at the next bar boundary"
- The transport completes the current loop and proceeds past the vamp's end marker

This makes "vamp until conductor cues end" a usable workflow without the conductor having to mash a button at a specific bar.

## MIDI recording

Coupled with backing tracks because the infrastructure overlaps:

- Capture MIDI input live while patches are playing
- Render the result to an audio file via offline render through the patch chain
- No piano-roll editor in v1.0 — the captured MIDI lives as a `.mid` file you can import to a DAW for editing
- Piano-roll editor is v1.x backlog

## What's deferred from v0.13.0

- **Full DAW MIDI import** — multi-track SMF → click + per-instrument tracks. v1.x.
- **Real-time time stretching** — for following a conductor with tracks. v2.0+.
- **Track stem mixing UI** in Perform mode beyond basic per-track mute. The full mixer is v0.10.0's Pit Mixer.

## Format note

Backing track support is the trigger for the **`.stardustshow/` bundle format migration** (also v0.13.0). Audio assets need to live alongside the show file. The single-JSON v1 format gets a one-shot migration to a folder bundle.
