> Stage right doesn't have line-of-sight to the conductor. The conductor cam puts the conductor on screen with the bar/beat/tempo/cue overlaid on top.

## What ships in v0.11.0

USB webcam input, video on screen, configurable overlay text, pop-out floating window for second-monitor display. That's the v1.0 scope — deliberately narrow.

## Capture

- **USB webcam only** — Pit reads from a UVC-compliant USB webcam attached to the host computer
- **Cross-platform** — uses [`nokhwa`](https://crates.io/crates/nokhwa) (or equivalent) for cpal-style abstraction
- **Resolution / framerate** — auto-detected; configurable in widget settings
- **Single camera** for v1.0

## Overlay

The widget composites text + meters over the video feed:

- **Song name**
- **Patch name**
- **Bar : beat** (from transport)
- **Tempo** (current BPM)
- **Clock** (running time)
- **Cue point** (current cue, if any)
- **Custom text** (markdown-rendered, per-show or per-song)

Each overlay element has position, font, size, color, and background opacity controls. Defaults are tuned for "readable over a typical conductor cam feed" — large, high-contrast, with a semi-transparent background.

## Two display modes

### Inline (default)

The widget sits in your Perform-mode layout like any other widget. Resizable. Useful when the conductor cam is part of your single-screen layout.

### Pop-out floating window

Right-click the widget → "Pop out." The conductor cam opens as a separate OS window that can be:

- Dragged to a second monitor
- Sized to fill that monitor
- Set to "always on top"
- Sent fullscreen

This is the common workflow for keyboardists with two screens: main screen has the Perform layout, second screen has only the conductor cam, fullscreen.

The pop-out window stays in sync with the inline widget — overlay updates happen in both.

## What's not in v1.0

The HANDOFF locks down v0.11.0 conductor cam at **USB webcam only.** Everything below is **v2.0+**:

- **RTSP / IP camera input** — needs a full RTSP client, codec coverage, network reliability work
- **NDI input** — needs the NDI SDK
- **Capture-card UI affordance** — HDMI capture cards present as webcam-class devices on most OSes, so they technically work, but the "this card needs a specific source mode" UX is deferred
- **Virtual webcam streaming output** — making Pit *be* a webcam to other apps

This isn't a permanent no — it's an explicit decision to ship the working narrow case first, validate it, then expand. RTSP and NDI together cover ~95% of pro AV setups; we want to do them well, not half-do them in v1.0.

## Performance

Video decode + overlay composite + display happens on a non-RT thread. The audio engine never touches the video path. A spike in video processing (camera resolution change, frame drop) can never cause an audio dropout.

## Privacy + permissions

- Camera access requires explicit OS-level permission (macOS: TCC; Windows: settings)
- The widget surfaces a clear indicator when the camera is active
- The camera is disabled by default; you opt in per-show
