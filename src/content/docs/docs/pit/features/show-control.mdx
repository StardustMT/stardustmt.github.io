> If Pit becomes the laptop running the show, the obvious next move is making Pit *run the show* — sound, lights, projection, automation, all triggered from one canonical timeline. That's Show Control.

This is a **v2.0+ vision page** — not a v1.0 commitment. It exists so the architectural decisions in v0.1.0 through v1.0 don't accidentally close the door.

## The protocols

Theatre show control isn't one protocol; it's a stack:

| Protocol | What it does | Target user |
|---|---|---|
| **MSC** (MIDI Show Control) | Trigger cues on lighting consoles, sound consoles, automation systems | Lighting / sound op |
| **OSC** (Open Sound Control) | General-purpose UDP messaging — Stage Manager apps, custom integrations | Custom workflows |
| **Art-Net / sACN** | Network DMX | Direct lighting control |
| **LTC** (Linear Time Code) | SMPTE timecode via audio | Sync to video / film playback |
| **MTC** (MIDI Time Code) | SMPTE timecode via MIDI | Sync to DAWs, video decks |

A v2.0+ Stardust Show Control surface implements all five as first-class engine sources/sinks, with a cue-list editor and a dedicated control panel.

## Why this fits Stardust

Pit already has:

- A precise transport with bar/beat/cue-point semantics
- An engine that respects allocation-free realtime constraints
- A patch graph with named sources and sinks
- A theatre-vocabulary UX (Show / Song / Patch — not Project / Track / Channel)

Adding MSC / OSC / DMX outputs is mostly engine plumbing — they slot into the existing transport. The hard part is the UX: a cue list editor that's usable by lighting and sound ops, not just keyboardists.

## Why this is v2.0+, not earlier

- v1.0 has to be *a great keyboard host first*. Show Control before the keyboard host is solid is feature creep.
- The cue-list UX is its own ~8–12 week effort.
- The hardware-integration testing matrix (real LX consoles, real automation systems) is large and expensive.
- Some of the protocol surface depends on the post-1.0 Stardust **Stage** runtime — having stage management and pit running on the same data model is what makes deep integration possible.

## What we're not committing to

- A QLab clone. QLab is an excellent app; we're not trying to replace it. Show Control in Stardust is for shows that want **one canonical timeline** running both the pit and the cues.
- DMX fixture libraries. If Pit needs DMX, it's at the protocol level — fixture knowledge belongs in [Stardust Lighting](/docs/ecosystem/roadmap/) (also speculative).
- A theatre ERP. Cue lists, yes. Production budgets, no.

## Adjacent work that has to land first

- **Plugin sandboxing** (v0.7.0) — out-of-process is a load-bearing decision the cue engine inherits
- **Transport state machine** (v0.8.0) — the timeline cue model rests on top of this
- **Click track editor** (v0.12.0) — cue points and timeline anchors get their first authoring surface here
- **Extension API** (v0.15.0) — third-party protocol implementations (proprietary lighting console protocols) live as extensions
